# i18n rules

The site ships in `en` (default) and `zh-Hans`. Translation is mandatory for every user-facing string.

## Source-code rules

- Every user-facing string in code must use `translate({ id, message })` (or `<Translate>`) from `@docusaurus/Translate`. After adding/changing strings, run `npm run i18n` and add the Chinese counterpart to `i18n/zh-Hans/code.json`.
- **Count-dependent strings use Docusaurus plural forms.** A string whose wording depends on a number authors its English `message` as a `singular|plural` template and resolves it at render with `usePluralForm().selectMessage(count, translate({ ... }))` â€” never string-concatenate a count with a label. The `zh-Hans` counterpart stays a single form (Chinese has no plural). Live users: `DataCard` labels and `PostMeta`'s word/view counts. `DataCard` runs _every_ label through `selectMessage`, so **all** `DataCard` `label` props must be `singular|plural` templates even when the two forms are identical (`'Category|Categories'`, `'Country|Countries'`) â€” a bare single form (`'Countries'`) is a latent inconsistency.
- **Never use a `theme.*` ID in custom code.** The `theme.*` namespace belongs exclusively to Docusaurus built-in strings (`@docusaurus/theme-classic` and friends) â€” they are scanned automatically from the framework's source. Reusing one in your own `translate()` call piggybacks on a framework-owned key and creates invisible coupling. Custom strings always go under a project-owned prefix (`pages.*`, `components.*`, `blog.*`, `data.*`).
- **Each i18n key may appear at most once in source.** If the same string is needed in multiple places, hoist the `translate()` call to a module-level `const` and reference the const â€” never paste the same `translate({ id, message })` literal twice. Duplicate occurrences are a refactor smell, not a feature.
- **Top-level prefixes are fixed at exactly five:** `theme` (Docusaurus built-ins only â€” see rule above), `pages` (page-specific strings under `src/pages/<page>`), `components` (reusable component strings under `src/components`), `blog` (blog area), `data` (static data files under `src/data`). Do not invent a sixth top-level prefix, and do not let a single component squat at the top level (e.g. `cookieConsent.*` was wrong; correct form is `components.cookieConsent.*`).
- **When you move a component or rename its folder, update its translation IDs to match.** The id's prefix should reflect _where the string is currently defined_, not its historical home. Example: a string in `src/components/Countdown` belongs under `components.countdown.*`, even if it used to live in `src/pages/_components/Countdown` and was named `home.countdown.*`. Stale-prefix ids accumulate as silent coupling; rename them in the same commit as the move.
- **Clean up orphan translations.** When you remove a `translate()` call (deleted component, removed feature), its entry in `code.json` becomes orphaned â€” `write-translations` does _not_ prune them. The canonical-rebuild workflow naturally drops them: backup `code.json` as `{key: {message}}` map â†’ `rm code.json` â†’ `npm run i18n` (regenerates with English fallbacks in canonical order, no orphans) â†’ walk fresh keys, override `message` with backed-up Chinese where present â†’ write back. Use this whenever you delete or rename source-side translation calls.
- **No `description` on custom `translate()` calls.** When you write a `translate({ id, message })` (or `<Translate id message>`) anywhere under our own code â€” `pages.*`, `components.*`, `blog.*`, `data.*` â€” do not add a `description` field. The id itself is the documentation; an extra description just bloats `code.json`. The exception is `theme.*` entries that come from Docusaurus's own source â€” their pre-existing `description` values stay untouched (we don't author those, we only translate the message).

## Key shape

- **3â€“5 segments, lowerCamelCase, no decoration.** A custom id is `<prefix>.<module>.<leaf>` at minimum (3 segments) and may nest one or two further levels for grouped UI (e.g. `pages.settings.item.theme.option.dark`, 6 segments is the absolute ceiling and reserved for option-of-item-of-group patterns). Segment names use lowerCamelCase; acronyms stay lowercased except as universally recognised tokens (`rss`, `json`, `atom`, `mbti`). Prefer full words over abbreviations (`systemStatus`, not `sysStatus`) so future-you doesn't have to guess.
- **Reuse the slot vocabulary; don't invent synonyms.** Standard leaf names â€” `title`, `description`, `modification`, `empty`, `placeholder`, `ariaLabel`, and action verbs `clear` / `reset` / `next` / `prev` / `play` / `pause` / `step` â€” have fixed meanings across the codebase. If a string fills one of those slots, use the slot name. Don't coin `heading`, `caption`, `blank`, `hint`, `a11yLabel`, etc.
- **Page header strings use the `title` / `description` / `modification` triplet.** Every `pages.<page>` that has a hero/header should expose its strings under those three exact leaves (modification is optional). Don't re-name them to `heroTitle`, `intro`, `updatedAt`. Pages without a header (e.g. `pages.home`) opt out entirely rather than substituting other words.
- **Datacard leaves are semantic, never numbered.** `pages.<page>.datacard.<concept>` (e.g. `categories`, `items`, `countries`, `years`). `label1` / `label2` / `value2` are banned â€” the next person reading the diff shouldn't have to open the page to find out what "label1" measures.
- **No stutter, no virtual single-key namespaces.**
  - Stutter: a leaf may not duplicate its parent segment. `components.problem.problem` âœ— â€” use `components.problem.statement` (or whichever slot word fits).
  - Virtual namespace: if a sub-namespace contains exactly one key, flatten it. `blog.sidebar.info.locationValue` âœ— â€” collapse `info` and rename to `pages.home.location` (this case also had the wrong prefix; both fixed in the same rename).
- **Generic single-word leaves only when the parent is descriptive enough.** `pages.changelog.empty` is fine because "the changelog's empty state" reads clearly. `components.markdown.title` was not fine â€” "the title of `components.markdown`" tells the reader nothing, because the component is `MDTitle` and the actual string is a greeting; rename to something specific like `components.markdown.aboutGreeting`.

## Translated content layout

Translated content lives under `i18n/zh-Hans/`:

- `docusaurus-plugin-content-docs/current.json` â€” sidebar/category labels
- `docusaurus-plugin-content-blog/` â€” translated `authors.yml`, `tags.yml`, blog options, and per-post MDX overrides
- `docusaurus-plugin-content-pages/<page>/` â€” translated MDX/markdown for custom pages (mirror the source filename)

`onBrokenLinks: 'throw'` is set, so missing translated targets will fail the build.
